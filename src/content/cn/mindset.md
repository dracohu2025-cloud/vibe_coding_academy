# Vibe Coding 心法：The Architect's Mind

> "2025年，编程不再是打字。编程是**编排 (Orchestration)**。"

很多人认为 Coding 就是学会 `for` 循环，学会 `if-else`。
但是在 Vibe Coding 的世界里，这只是表象。真正的核心是你的**心智模型 (Mindset)**。
当 AI (Claude, Gemini, GPT) 的智商超越 99% 的人类时，你的角色不再是“码农”，而是**架构师 (Architect)**。

---

## 1. 身份认同：从 Builder 到 Commander

甚至在 2024 年，我们还在说 "Co-Pilot" (副驾驶)。
但在 2025 年，这个概念已经过时了。你不再需要一个副驾驶，你需要指挥一支**军队**。

### 你的 AI 小队 (The Squad)

不要只用一个 AI。要把它们视为具有不同性格的**员工**：

* **Claude Opus 4.5 (首席架构师)**：
  * *性格*：严谨，深思熟虑，能看到长远的隐患。
  * *用途*：系统设计，复杂 Bug 排查，重构建议。当你不知道该怎么做时，问它。
* **GPT-5.2 (高级工程师)**：
  * *性格*：执行力强，速度快，通晓所有流行框架。
  * *用途*：生成 [[React]] 组件，写 SQL 查询，转换数据格式。当你需要代码实现时，找它。
* **[[Gemini 3.0]] Pro (资料库管理员)**：
  * *性格*：记忆力无限，吞噬万卷文档。
  * *用途*：阅读 100 页的 API 文档，分析 10MB 的日志文件。当你需要查找信息时，派它去。

**Vibe 原则**：如果你自己写了超过 50 行的样板代码，你就已经输了。你应该是在 Review 代码，而不是 Write 代码。

---

## 2. 提示工程：是对话，不是命令

新手对 AI 吼叫："写一个登录页面"。
大师与 AI 对话："我们来设计一个登录流，考虑到安全性..."

### 上下文倾倒 (Context Dumping)

AI 犯错的唯一原因就是**它不知道你知道的信息**。
在 2025 年，我们有超大的 Context Window (200K+ token)。

* **不要节省 Token**。把你的 `schema.prisma`，你的 `types.ts`，甚至你的产品需求文档 (PRD)，全部扔给它。
* **Prompt 模板**：
    > "你是一名高级 [[React]] 工程师。基于我提供的 `<DesignSystem>` 组件库和 `<DatabaseSchema>`（见附件），请实现用户个人中心页面。注意：必须遵循我的 [[ESLint]] 规则（见附件）。"

### 苏格拉底式编程 (Socratic Coding)

不要让 AI 直接给答案。让它**先思考**。
> "在我让你写代码之前，先列出实现这个功能的 3 种方案，并分析它们的优缺点。推荐哪一种？为什么？"

这能迫使模型进入思维链 (Chain of Thought) 模式，产出质量高出 10 倍的代码。

---

## 3. 视觉即功能 (Visuals First)

在传统软件工程中，UI 是最后做的“皮肤”。
在 Vibe Coding 中，**UI 是灵魂的载体**。

* **美学驱动开发 (ADD)**：先用 v0.dev 生成一个极其漂亮的界面。看着这个界面，你会更有动力去实现背后的逻辑。
* **Drip (风格)**：你的代码本身也应该好看。对齐的缩进，清晰的变量名，合理的空行。当你打开一个文件，它应该像一篇排版精美的文章。
  * *坏的味道*：嵌套 5 层的 `if-else`。
  * *好的 Vibe*：扁平化的 Early Return。

---

## 4. 极速反馈 (Instant Feedback)

Vibe 的天敌是**等待**。

* 如果 `npm run dev` 启动超过 3 秒，换 Vite。
* 如果 CI 部署超过 2 分钟，优化 Dockerfile。
* 如果数据库查询慢，加 Redis。

你想要的是**心流 (Flow)**。任何打断心流的延迟都是不可接受的。你的工具链必须像你的思维一样快。

## 总结

Vibe Coding 不是关于你会多少个 API。
它是关于你如何**调动硅基智能**，如何**追求极致美学**，以及如何**保持心流状态**。
当你掌握了这套心法，你就不再是一个在键盘上打字的工人，你是一个在数字世界中编织现实的法师。
